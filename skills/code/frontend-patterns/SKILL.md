# Moodle Frontend Patterns

## AMD Module Structure

JavaScript in Moodle uses ES2015 modules compiled to CommonJS format, loaded via RequireJS.

**File locations:**
- Source: `amd/src/` — this is what you write.
- Built: `amd/build/` — generated by Grunt, do not edit.
- In development mode Moodle reads from `amd/src/` directly when `$CFG->cachejs = false`.

**Module naming convention:** `componentname/modulename` (e.g., `mod_pluginname/booking`).

**Standard module structure:**

```javascript
// amd/src/booking.js

/**
 * Booking management for mod_pluginname.
 *
 * @module     mod_pluginname/booking
 * @copyright  2026 Your Name <your@email.com>
 * @license    http://www.gnu.org/copyleft/gpl.html GNU GPL v3 or later
 */

import Ajax from 'core/ajax';
import Notification from 'core/notification';
import Templates from 'core/templates';

/**
 * Initialize the booking interface.
 *
 * @param {number} cmid - The course module ID.
 */
export const init = (cmid) => {
    document.addEventListener('click', (e) => {
        const btn = e.target.closest('[data-action="book-slot"]');
        if (!btn) {
            return;
        }
        e.preventDefault();
        bookSlot(cmid, parseInt(btn.dataset.slotid, 10));
    });
};

/**
 * Book a slot via AJAX.
 *
 * @param {number} cmid - Course module ID.
 * @param {number} slotId - The slot ID to book.
 */
const bookSlot = async(cmid, slotId) => {
    try {
        const result = await Ajax.call([{
            methodname: 'mod_pluginname_create_booking',
            args: {cmid, slotid: slotId},
        }])[0];

        // Refresh the slot list after booking.
        await refreshSlotList(cmid);
    } catch (err) {
        Notification.exception(err);
    }
};
```

**Include the module from a Mustache template:**

```mustache
{{#js}}
require(['mod_pluginname/booking'], function(Booking) {
    Booking.init({{cmid}});
});
{{/js}}
```

Or from PHP using `$PAGE->requires->js_call_amd()`:

```php
$PAGE->requires->js_call_amd('mod_pluginname/booking', 'init', [$cm->id]);
```

## Cache Purging After JS Changes

**Always purge caches after changing AMD files:**

```bash
php admin/cli/purge_caches.php

# In DDEV:
ddev exec php admin/cli/purge_caches.php
```

In production, also run the Grunt build:

```bash
grunt amd --root=mod/pluginname
```

Without cache purging, Moodle will serve stale JS even if the source file changed.

## AJAX via External Functions

Call registered external functions from JavaScript using the `core/ajax` module.

```javascript
import Ajax from 'core/ajax';

// Single call — returns a Promise.
const promise = Ajax.call([{
    methodname: 'mod_pluginname_get_slots',
    args: {cmid: 42},
}])[0];

promise.then((slots) => {
    // slots is the array returned by execute_returns().
}).catch(Notification.exception);

// Multiple calls in one round trip:
const [slotsPromise, bookingsPromise] = Ajax.call([
    {methodname: 'mod_pluginname_get_slots', args: {cmid: 42}},
    {methodname: 'mod_pluginname_get_bookings', args: {cmid: 42}},
]);
```

**Prerequisites for AJAX calls to work:**
1. The function must be registered in `db/services.php` with `'ajax' => true`.
2. A version bump and upgrade must have run after adding/changing `services.php`.
3. The function must exist in `mdl_external_functions` table.

**Debugging: If you get `invalidrecord` errors**, the function is not registered. Check:
```sql
SELECT * FROM mdl_external_functions WHERE name = 'mod_pluginname_get_slots';
```
If no row exists, the services.php registration did not take effect — bump the version and re-run the upgrade.

## Mustache Templates

Templates live in `templates/`. File name maps to `componentname/templatename`.

```
templates/
├── view.mustache          → mod_pluginname/view
└── slot_list.mustache     → mod_pluginname/slot_list
```

**Basic template with string helper:**

```mustache
{{!
    @template mod_pluginname/slot_list

    Template for displaying the list of available slots.

    Context variables:
    - slots: array of slot objects
      - id: int
      - name: string
      - available: bool
}}
<div class="mod-pluginname-slots" id="mod-pluginname-slot-list">
    {{#slots}}
    <div class="slot-item" data-slotid="{{id}}">
        <span class="slot-name">{{name}}</span>
        {{#available}}
        <button class="btn btn-primary" data-action="book-slot" data-slotid="{{id}}">
            {{#str}}booknow, mod_pluginname{{/str}}
        </button>
        {{/available}}
    </div>
    {{/slots}}
    {{^slots}}
    <p>{{#str}}noslots, mod_pluginname{{/str}}</p>
    {{/slots}}
</div>
{{#js}}
require(['mod_pluginname/booking'], function(Booking) {
    Booking.init({{cmid}});
});
{{/js}}
```

**Render from PHP:**

```php
$templatecontext = [
    'cmid'  => $cm->id,
    'slots' => array_values($slots), // Must be indexed array, not associative.
];
echo $OUTPUT->render_from_template('mod_pluginname/slot_list', $templatecontext);
```

## Mustache Partials and Context Scope

When using `{{> mod_pluginname/partial}}`, the partial receives **only the current context scope**. It cannot access parent-level variables.

**Wrong — partial cannot see `cmid` from the parent:**

```mustache
{{! view.mustache }}
{{#slots}}
    {{> mod_pluginname/slot_item}}   {{! slot_item cannot access cmid }}
{{/slots}}
```

**Correct — flatten the context in export_for_template():**

```php
// In your renderable class:
public function export_for_template(\renderer_base $output): array {
    $slots = [];
    foreach ($this->slotlist as $slot) {
        $slots[] = [
            'id'    => $slot->id,
            'name'  => $slot->name,
            'cmid'  => $this->cmid,  // Promote cmid into each slot item.
        ];
    }
    return [
        'cmid'  => $this->cmid,
        'slots' => $slots,
    ];
}
```

This is the correct pattern: flatten all data needed by each partial into that partial's own context object in `export_for_template()`.

## Form Change Checker Suppression

Moodle's form change checker shows a "You have unsaved changes" dialog when the user navigates away from a modified form. If your plugin has its own confirmation dialogs (e.g., a "Cancel booking?" dialog that submits a form), this causes a duplicate dialog.

Suppress Moodle's dialog on your form element:

```html
<form method="post" data-no-submit-warning="1">
```

Or in PHP `moodleform`:

```php
$mform->setAttributes(['data-no-submit-warning' => '1']);
```

## Delegated Event Handlers

When AJAX calls replace page content, elements you attached `addEventListener` to are destroyed and re-created. If you re-attach handlers after each AJAX response, you accumulate duplicate handlers.

**Wrong — attaches duplicate handlers after each refresh:**

```javascript
const refreshSlots = async() => {
    const html = await fetchNewHtml();
    container.innerHTML = html;
    // This re-attaches, creating duplicates.
    document.querySelectorAll('[data-action="book-slot"]').forEach(btn => {
        btn.addEventListener('click', handleBooking);
    });
};
```

**Correct — attach once to a persistent parent:**

```javascript
// Attach once during init, to a container that persists across AJAX refreshes.
document.getElementById('mod-pluginname-slot-list').addEventListener('click', (e) => {
    const btn = e.target.closest('[data-action="book-slot"]');
    if (!btn) {
        return;
    }
    handleBooking(btn);
});

// AJAX refresh only updates innerHTML; the handler on the container still works.
const refreshSlots = async(cmid) => {
    const slots = await Ajax.call([{methodname: 'mod_pluginname_get_slots', args: {cmid}}])[0];
    const {html} = await Templates.renderForPromise('mod_pluginname/slot_list', {slots});
    document.getElementById('mod-pluginname-slot-list').innerHTML = html;
};
```

## RequireJS Debugging

If an AMD module fails to load:

1. **Purge caches first:** `php admin/cli/purge_caches.php`
2. **Check the module path.** The file must exist at `mod/pluginname/amd/src/modulename.js`.
3. **Check the browser console** for RequireJS errors.
4. **Moodle silently falls back** to core JS bundles if the path is wrong — it does not return a 404. The symptom is that the page loads but your module code never runs.
5. **Enable JS debugging** in Site Admin > Development > Debugging, or set `$CFG->cachejs = false` in `config.php`.

## Fetching Language Strings in JavaScript

Use `core/str` to retrieve translated strings asynchronously:

```javascript
import {get_string as getString, get_strings as getStrings} from 'core/str';

// Single string:
const label = await getString('booknow', 'mod_pluginname');

// Multiple strings in one request:
const [book, cancel] = await getStrings([
    {key: 'booknow', component: 'mod_pluginname'},
    {key: 'cancel',  component: 'mod_pluginname'},
]);
```

The strings must be defined in `lang/en/pluginname.php` or the request will return the raw key.
